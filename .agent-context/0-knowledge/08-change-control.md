# Change Control

Lyra covers **two types of changes**:

1. **Running audit of changed fields** — who changed what, when (audit log)
2. **Changes that users care about** — changes exceeding a certain threshold (change management)

## Principles

- User can **switch type 2 CC on or off**. Having it switched on creates a level of admin.
- Purpose of the functionality:
  - Give a **warning of potential impact**
  - Give a **retrospective visual** of the changes over time
  - Understand **why things changed** both in aggregation and retrospectively for future planning purposes

## Audit Log

Each field change is captured on the change audit log. Changes are flagged and filterable:

| Change Type | Description |
|-------------|-------------|
| System generated | Automatic system changes |
| Task relationship | Changes made as a result of a dependency (e.g. predecessor) |
| Baseline & re-baseline | Baseline events |
| Text changes | e.g. task name changes |
| Dependency | Dependency-related changes |

- Audit log is exportable to **MS Excel**

## Approval Threshold

The approval threshold is set up in the change screen in **Settings**:

- Can be applied to **any attribute** (date changes is the default)
- The period or calendar date can then be set
- **Different thresholds** can be set for different projects

### When the Threshold is Breached

The following actions happen:

1. Change log item will be created, approval column = **'Required'**
2. CR form will be generated and auto-populated (optional)
3. Task in the plan will have:
   - Status change to **Red**
   - A **CC flag** will appear
   - Text will change to **italic**
   - Font and gantt will appear **paler**
4. Risk or issue will be logged with **AI-generated text** — user to update commentary for reason

## Change Workflow

Change log item status defaults to **'Required'** when threshold has been breached. User has the following options:

### Option 1: Ignore

- Remove CC flag — this will **revert RAG and formatting**
- Updates the Change log item status to **'User Override'**

### Option 2: Work Through Change Process

1. Comment on reason for change
2. Seek approval decision (outside Lyra), supported by CR form generated by Lyra (optional)
3. Log approval decision:
   - **Approved** will update the plan: re-baselining the task to Green
   - User can upload evidence of approval and/or comments to the individual change item

## Change Log — Breach & Audit

| Field | Source |
|-------|--------|
| Status | See status values below |
| Timestamp | Automated |
| User changed | Automated |
| Column | Populated from plan |
| Previous Value | Populated from plan |
| New Value | Populated from plan |
| Change Reason | Populated from plan |
| Has Deps | Automated |
| Total Time Change | Automated count |
| Breach Type | Automated |
| Total Changes | Automated |
| Approval Timestamp | Automated: Date flag removed |
| Task Name | Automated |
| Parent Task Name | Automated |

### Status Values

| Status | Description |
|--------|-------------|
| Approved | Automated: Date flag removed |
| User overwrite | Approval not required |
| N/A | For audit log, not breaches |
| Awaiting approval | Default status for new breaches |
| Rejected | — |

> Ideally the user doesn't edit much here. It's a low-admin, trusted, selective audit of history that users can interrogate. The status will have to be updated manually other than where described (awaiting approval & approved) which can be overwritten.

Export to Excel/other available.

## Scope

- **In scope**: Baselined tasks. Tasks may be baselined in isolation. If a parent is baselined then the children are also baselined. One workstream can baseline ahead of another — this is usually the case, with more complex workstreams baselining later and workstreams closer to go-live baselining later.
- **Out of scope**: Pre-baselined row items

### Change Control Flow (Post-Baseline)

```
Date is changed in baselined plan and breaches threshold set
        │
        ▼
Lyra Automation:
  - Format changed to show potential delay on all impacted tasks
  - CC flag applied
  - Status flips to Red
  - Change log entry created with commentary added
  - Optional CR form generated
  - Notification to vertical owner of breach & Planner
        │
        ▼
CR FORMS ──> Outside Lyra: Change Board / Vertical owner to review
        │
        ▼
Manual intervention?
  ── No ──> User removes flag on plan screen & updates date
            Dialogue box: Lyra updates change log to approved &
            reverts all formatting. Undo options for previous 2 actions
  ── Yes ──> Plan is baselined following board approval
             New baseline for impacted tasks
             Commentary updated to explain cause of underlying breach
```
